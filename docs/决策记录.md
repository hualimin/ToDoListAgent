## ADR-012 · 子系统 A 拆分为 A1/A2/A3，顺序 A2→A1→A3（2026-06-26）
- **背景**：子系统 A（待办编排）含多个特性簇（多模态录入+AI解析 / 看板+视图 / 智能编排），单 spec 装不下。
- **决定**：拆 3 个独立子项目，按 **A2→A1→A3** 顺序做（A2 先做看板+视图，纯前端，顺带修地基 board_order/withinRange；A1 再加多模态+AI录入；A3 最后做需 A1+A2 的智能编排）。历史可视化（月历热力/年趋势）单独再拆（不进 A2）。
- **后果**：每个子项目独立「规格→计划→实现」循环。

## ADR-013 · A2 看板交互定稿（2026-06-26，经 6 版原型迭代）
- **背景**：A2（任务看板+多粒度视图）经 6 版交互原型（docs/prototypes/a2-board-demo-v1~v6.html）与用户多轮确认。
- **决定**（A2 最终形态）：
  - **状态板**：5 状态列（待办/进行中/完成/搁置/延期）**纵向铺开**（窄屏垂直滚动）；拖动改状态、列内拖调序；**跨列拖动弹确认**（确认才改状态+样式，取消回滚，防误拖；同列内调序不确认）。
  - **列表**：地基列表升级，每项带状态图标+色条+紧急度+时间；长标题两行省略；点卡进**可编辑详情**。
  - **月历**（替换原"时间板"）：固定格子尺寸（高约 74px），任务文字过长 `text-overflow:ellipsis` 单行截断，每格最多 2 条（重要优先）+「+X 更多」；**只读**（不改日期）；右上角图标切状态板/月历/列表。
  - **点某天展开当天全部**（含已完成历史）。
  - **月历年/月/日跳转组件**（选年月日直达某天，或回今天）。
  - **改日期走详情抽屉日期选择器**（不在月历/板上拖——日期是开放范围，拖列不实用）。
  - 视图切换器在**顶部独占行**（不遮挡内容）。
  - 触摸拖拽用 **dnd-kit**（原生 HTML5 拖拽触屏不支持）；全程 4 主题感知；修地基 `board_order`（float 中点重排）+ `withinRange`（今日/本周语义=今天/本周内含逾期）。
- **后果**：A2 范围明确，UI 已原型验证，可直接进规格→计划→实现。

## ADR-011 · 前端 UI 为"主题化组件系统"，运行时可切换 4 套皮肤（2026-06-25）
- **背景**：用户从 5 套原型中选定 4 套（01 纸本日记 / 02 瑞士极简 / 04 明快活力 / 05 自然手作），希望运行时随时切换，而非只做一套。
- **决定**：
  - 前端做**一套主题化组件系统**：同一套组件（功能/交互/信息结构一致），通过**主题变量**（颜色/字体/圆角/质感）+ 允许**小布局差异**（印章标签/圆点/渐变 hero 等各主题特色）呈现 4 种皮肤。
  - 设置页提供**主题切换**，用户偏好持久化到本地。
  - **默认主题 = 05 自然手作**。
  - 功能对齐：4 套皮肤共享相同功能（周历日期带、完成趋势、范围筛选、任务详情页、反思聚类、学习由浅入深节点）。01 纸本需补齐早期缺失的这些功能。
  - 暗色模式**本期不做**（03 墨夜暗色未入选，后期可补暗色变体）。
- **后果**：
  - 工程更复杂（主题变量层 + 条件渲染小差异），但用户体验与可维护性更好，切换无大布局跳动。
  - 前端地基计划需相应调整：UI 任务从"占位 UI"升级为"主题化组件系统（4 皮肤）+ 切换器"。
  - 设计 token（CSS 变量/Tailwind 主题）是核心抽象，新增皮肤成本低。
- **原型**：见 `docs/prototypes/`（4 套 `*-style.html` + `index.html` 总览）。

## ADR-010 · 前端在 Windows Node 运行，后端在 WSL Python 运行（2026-06-25）
- **背景**：WSL 无 node（仅有泄漏进来的 Windows npm，无法执行）；Windows 本机有 Node v24。前后端需不同运行时。
- **决定**：后端 Python 跑 WSL；前端 Node/Vite/Capacitor 跑 Windows（Git Bash 直接执行，无需 wsl.exe）。二者共用同一份项目文件（`e:/` ↔ `/mnt/e/`）。
- **后果**：环境说明需区分；端到端联调时前端访问 `http://localhost:8000` 调 WSL 里的后端（同机可互通）。

## ADR-009 · AI 统一走 OpenAI 兼容接口（2026-06-24）
- **背景**：需支持多家厂商（OpenAI/Claude/DeepSeek/智谱/通义/Kimi），且每功能可独立配 API。
- **决定**：agent_registry 统一用 OpenAI 兼容（base_url + api_key + model）调用，按功能角色路由；个别厂商差异在适配层抹平。
- **后果**：配置统一、用户切换厂商成本低；少数特殊能力（如部分工具调用语义）需按 provider 分支处理（见规格 R2）。

## ADR-008 · 密钥只存 config 文件，绝不进 DB（2026-06-24）
- **背景**：项目要开源，API Key 与个人敏感信息必须可剥离。
- **决定**：所有密钥/SMTP/webhook/auth 令牌只存 `config/secrets.local.json`（gitignored）；App 设置页经后端原子写入该文件；DB 永不存密钥。
- **后果**：开源剥离 = 删 `config/secrets.local.json` + `server/data/*.db`；需脚本/CI 校验仓库零敏感信息（见规格 R3）。

## ADR-007 · 通知渠道 MVP = 邮件 + App内消息 + Webhook（2026-06-24）
- **背景**：提醒是核心；非 App 内渠道需用户手动配。
- **决定**：MVP 用邮件(SMTP) + App内消息 + Webhook(Bark/钉钉/飞书/企微)；不做 FCM/APNs。
- **后果**：自部署简单、覆盖广；App 关闭时靠邮件/Webhook 保底，App内消息需开 App 才见。真·系统推送列为增强。

## ADR-006 · 后端存储用 SQLite（可切 Postgres）（2026-06-24）
- **背景**：后端必须持久化提醒队列/调研结果/配置，但无需跑 DB 服务器。
- **决定**：后端用 SQLite 单文件（SQLAlchemy 抽象），未来可切 Postgres。
- **后果**：零配置自部署；多用户升级时切 Postgres 即可，ORM 层无大改。

## ADR-005 · 数据架构：本地优先（2026-06-24）
- **背景**：用户质疑"为啥要数据库"，倾向客户端保存；但邮件/Webhook 提醒与后台调研必须服务端。
- **决定**：业务数据（任务/反思/学习）以设备端 SQLite 为源；后端只存"提醒队列+调研结果+密钥"，做定向同步（仅带提醒项上行、仅调研结果下行）。
- **后果**：隐私(数据在设备)、离线可录、无需部署 DB 服务器；代价是写一套轻量定向同步（单用户，幂等 upsert，冲突极小）。

## ADR-004 · 后端技术栈 = Python + FastAPI（2026-06-24）
- **背景**：B/C 子系统大量 AI/调研/聚类/RAG；提醒调度、Agent 调用都在后端。
- **决定**：Python 3.11 + FastAPI + SQLAlchemy + APScheduler。
- **后果**：AI/调研生态最强；前后端跨语言（前端 TS），类型不共享（用 OpenAPI 生成前端类型缓解）。

## ADR-003 · 用户/部署模型 = 单用户 MVP，预留 user_id（2026-06-24）
- **背景**：个人自用，但开源后他人可能自部署；未来或有多账号需求。
- **决定**：MVP 单用户（user_id=1），所有表预留 user_id 字段与接口约定，平滑升级多用户。
- **后果**：当前无需注册/多租户/配额复杂度；升级多用户时数据模型基本不动。

## ADR-002 · APP 形态 = Web 技术栈 + Capacitor 套壳（2026-06-24）
- **背景**：移动端优先；开发流程是 image2→codex→figma 的 Web 组件工作流。
- **决定**：React + TS + Vite + Capacitor（必要时 Tauri 桌面）。
- **后果**：契合组件工作流、移动端友好、系统级能力(语音/后台/相册)经 Capacitor 获得；移动推送在 MVP 阶段受限(见 ADR-007)。

## ADR-001 · 构建顺序 = 共享地基 → 子系统 A → B → C（2026-06-24）
- **背景**：项目含 3 个相对独立子系统 + 共享底座，单份规格装不下。
- **决定**：先建共享地基，再依次做 A(待办编排)、B(细小问题反思)、C(主动学习路径)，各自独立「规格→计划→实现」。
- **后果**：地基一次到位、返工最少；首个可用功能是待办编排。
